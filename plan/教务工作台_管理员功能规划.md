# 教务工作台（管理员端）功能规划

> **定位**：面向毕设演示与后续迭代的**产品级说明**，与现有后端 `User.role = admin`、错误案例管理中的管理员分支对齐，供后期实现时对照拆解任务。  
> **文档版本**：2026-04-30（2026-04-30 文档同步）  
> **状态**：**M1～M3 已落地**（`/admin` 工作台、用户列表/详情、毕业进度穿透、学业预警为「学生下次登录弹窗一次」、论坛治理含恢复）；**M4/M5 仍为规划中**（错误案例管理 UI、课表只读 Tab、`PATCH /api/admin/users/{student_id}`、审计与导出等，见第 5～6 节表格）。

---

## 1. 背景与目标

### 1.1 为什么要单独做「教务工作台」

- 学生端强调**个人学业路径**（我的课表、我的毕业进度、我的发帖等）。
- 教务/运营视角需要**跨用户、可筛选、可解释**的只读或有限治理能力，界面以**列表 + 详情 + 统计**为主，与学生端信息架构区分开，答辩时边界清晰。

### 1.2 与现有代码的衔接（实现前必读）

| 现状 | 说明 |
|------|------|
| `users.role` | 已支持 `student` / `admin`（见 `backend/app/models/user.py`）。 |
| 注册流程 | 默认创建 `student`；**无**公开「自提权为 admin」接口，首个管理员建议脚本或受控内部流程创建。 |
| 登录与鉴权 | JWT + `GET /api/auth/me` 可返回用户信息（含 `role`，见 `backend/app/schemas/auth.py`）。 |
| 错误案例 | `ErrorCaseService` 已对 `admin` 与「本人数据」做区分（见 `backend/app/services/error_case.py`），可作为**权限模型参考实现**。 |
| 学号查询 | 学生端接口必须收敛为「本人或 admin 可查」，否则普通学生可越权查看任意学号进度。 |

### 1.3 设计原则（建议写进技术文档/答辩口径）

1. **最小权限**：默认只读查询；删改、升降级等高风险操作单独开关或二期再做。  
2. **前后端一致**：前端隐藏菜单不能代替鉴权；所有管理接口必须校验 `current_user.role == "admin"`。  
3. **可审计（推荐二期）**：敏感查询（如查看某学生完整进度）记录操作者、时间、目标学号。  
4. **数据不混淆**：管理员查看「某学生进度」时，UI 明确展示**被查看学号**，避免与「自己的学生账号」进度混同（若管理员同时也是学生账号则尤其重要）。

## 2. 角色与账号策略

### 2.1 角色定义

| 角色 | 说明 |
|------|------|
| `student` | 默认；访问个人相关接口与公共资源。 |
| `admin` | 教务工作台；可访问本规划中的管理类接口与前端 `/admin`（或等价路由）。 |

### 2.2 管理员账号创建（后期实现时择一或组合）

- **推荐（安全）**：本地脚本 / 初始化命令，在库中无 admin 或带环境变量密钥时创建首个管理员；学号占位如 `admin001`（需满足现有 `username = student_id` 约束）。  
- **可选**：受 `ADMIN_BOOTSTRAP_TOKEN` 保护的一次性 HTTP 接口（仅内网或部署时启用）。  
- **不推荐**：注册接口开放 `role` 字段由客户端自选。

### 2.3 与「学生数据」的关系

- 管理员账号在数据模型上仍是 `User`，可绑定 `major` 等字段。  
- **业务约定**：教务账号可不参与选课逻辑；若参与，需在需求上写明「管理员是否占用学生名额」——建议毕设阶段**不参与**，减少边界情况。
- 「返回学生端」仅表示用当前管理员账号进入普通前台路由查看学生端页面效果，并不代表管理员账号一定拥有完整学生学业数据。为避免迷路，学生端若当前登录人为 `admin`，应提供明显「返回教务工作台」入口。

### 2.4 演示账号与数据前置（P0 落地要求）

- 管理员账号通过脚本或受控初始化创建，不开放普通注册自选 `role=admin`。
- 已有场景学生账号需保证可登录、绑定专业、年级、已修课程、课表与时间规划数据齐全，便于管理员列表和毕业进度穿透演示。
- 演示数据脚本应尽量幂等：重复执行时更新既有账号与演示记录，不生成难以追踪的重复管理员。
- 管理员账号不写入学生课程成绩与时间规划，避免进入毕业进度统计样本。

## 3. 功能模块详述

以下按**教务工作台菜单**组织，每项标注建议优先级（P0 答辩最小闭环 / P1 完整工作台 / P2 加分项）。

### 3.1 工作台首页（Dashboard）— P0 / P1

**目标**：一屏了解系统运行概况。

| 能力 | 说明 | 优先级 |
|------|------|--------|
| 核心指标卡片 | 注册用户总数、按专业分布（可选）、近 7 日活跃（若有埋点或登录日志则 P1，否则占位） | P0 可简化为「总用户数」 |
| 快捷入口 | 跳转「用户与进度」「内容治理」 | P0 |
| 系统健康 | 只读展示后端 health（可选，P1） | P1 |

**数据依赖**：用户表计数、专业表维度统计；无需改学生核心业务表结构即可起步。

---

### 3.2 用户与账号管理 — P0 / P1

**目标**：按学号/专业/角色筛选用户，支持运营排查与演示「看见所有账号」。

| 能力 | 说明 | 优先级 |
|------|------|--------|
| 用户列表 | 分页；列：学号、姓名（若有）、专业、角色、`is_active`、注册时间 | P0 |
| 搜索与筛选 | 学号精确/模糊、专业代码、角色、是否启用 | P0～P1 |
| 用户详情抽屉/页 | 展示基础档案 + **入口：查看该用户毕业进度（只读）** | P0 |
| 发送学业预警 | 对指定学生发送一条只读预警提醒，学生下次登录时以弹窗展示一次 | P1 |
| 启用/禁用 | 更新 `is_active`（需后端接口 + 审计日志更佳） | P1 |
| 角色变更 | 将某用户升为 `admin` 或降为 `student` | P2（高风险，可答辩时不演示） |

**权限**：仅 `admin`。  
**安全**：列表接口禁止无分页的全表 dump 到前端；导出 CSV 若做则 P2 且需单独评审。

**P0 接口返回建议**：

- 列表接口返回 `{ items, total, page, pageSize }`，前端表格按分页渲染。
- 用户项统一返回 `studentId`、`name`、`majorName`、`majorCode`、`grade`、`role`、`isActive`、`createdAt`、`avatarUrl`。
- 搜索参数支持 `q`（学号/姓名模糊）、`majorCode`、`role`、`isActive`，先满足演示排查，不做导出。

**学业预警落点（当前实现建议）**：

- P1 阶段使用独立 `academic_warnings` 表：管理员发送预警时写入目标学生待展示记录。
- 学生登录成功后，`/api/auth/login` 返回 `pendingAcademicWarnings`，前端弹窗展示；后端同时写入 `shown_at`，确保只弹一次。
- 预警只允许管理员创建，不进入 `time_plan_events`，学生不能伪造或在时间规划中编辑删除。

---

### 3.3 毕业进度「教务视图」— P0（核心演示）

**目标**：回答导师/评委问题——「教务能否掌握全班/全系毕业达标情况？」

| 能力 | 说明 | 优先级 |
|------|------|--------|
| 按用户查看进度 | 与现有学生「个人毕业进度」**同一套业务规则与数据结构**，仅换「目标 user_id / student_id」 | P0 |
| 列表穿透 | 从用户列表点「查看进度」进入；URL 建议带 `student_id` 或内部 `user_id`（与后端约定一致） | P0 |
| 专业维度列表（可选） | 某专业下学生进度一览：学号、完成度摘要、风险提示（规则与前端学生端一致即可） | P1 |
| 只读 | 不在此界面修改学分或规则；若未来有「教务更正」需单独模块与审批流 | 规划内默认只读 |

**后端要点（后期实现时）**：

- 复用现有「毕业进度」服务层，增加参数 `target_user_id`，在路由层校验：  
  - `admin`：允许查询任意用户；  
  - `student`：仅允许查询自己（保持现状）。  
- 避免复制一套计算逻辑导致两处不一致。
- 管理端建议提供 `/api/admin/users/{student_id}/academic-info`，直接复用学生端 `AcademicInfoResponse`，方便前端复用 `AcademicInfoCard`。

---

### 3.4 课表与学业数据（只读）— P1

**目标**：辅助答疑——某学生课表来源、学业查询结果是否与进度一致。

| 能力 | 说明 |
|------|------|
| 按学号查看课表 | 只读；接口层与 3.3 相同模式：`admin` + `target_user_id` |
| 学业相关只读接口 | 与现有学生接口对齐，统一走管理鉴权包装 |

若毕设时间紧，**可合并到 3.3 用户详情中的 Tab**，不必单独一级菜单。

---

### 3.5 内容治理 — P0 / P1

与现有后端能力对齐，避免重复造轮子。

| 子模块 | 能力 | 与现有代码关系 |
|--------|------|----------------|
| AI 错误案例库 | 列表全量或按条件筛选；详情；删除/归档（以当前 `error_case` 路由能力为准扩展） | 已有 admin 与所有者区分逻辑 |
| 论坛治理 | 帖子列表、按标题/作者/专业筛选、隐藏/删除违规帖子、取消隐藏恢复展示 | 复用 `ForumTopic.status` 做软删除/隐藏，P1 可先不做操作原因 |

**优先级**：错误案例管理 P0（已部分具备后端基础）；论坛治理 P1。

**论坛治理 P1 边界**：

- 管理端列表应独立于学生端论坛列表，允许管理员看到更多治理字段（作者学号、作者姓名、状态、评论数、点赞数、浏览量、创建时间）。
- 「隐藏/删除」优先实现为软删除：更新 `status=deleted`，学生端正常列表不再展示；物理删除附件与评论属于 P2。
- 「取消隐藏」将 `status` 恢复为 `normal`，学生端列表重新展示；与隐藏对称，同样需要二次确认。
- 前端操作必须二次确认，并提示该操作会从学生端隐藏帖子。

---

### 3.6 统计与导出 — P2

| 能力 | 说明 |
|------|------|
| 毕业达标率按专业 | 基于只读聚合，规则与进度计算一致 |
| 导出报表 | Excel/PDF；注意隐私与脱敏，毕设可选 |

---

### 3.7 操作审计 — P2（加分）

| 能力 | 说明 |
|------|------|
| 审计表 | 记录管理员 id、动作类型、目标资源、时间、请求 id |
| 审计页 | 简单表格筛选，仅 `admin` 可见 |

答辩话术：「当前最小实现以接口鉴权为主，审计为演进路线。」

## 4. 前端信息架构（UI）建议

### 4.1 路由与布局

- **路径前缀**：`/admin`（推荐），与主站 `/` 或 `/app` 并列。  
- **布局**：左侧导航 + 顶栏（当前管理员学号/姓名、退出）；风格可偏简洁表格风，不必与学生端完全一致。  
- **鉴权**：进入 `/admin` 前检查 `me.role === 'admin'`，否则跳转 403 页或回首页。

### 4.2 页面清单（与第 3 节对应）

| 路由示例 | 页面名称 | 模块 |
|----------|----------|------|
| `/admin` | 工作台首页 | 3.1 |
| `/admin/users` | 用户与账号 | 3.2 |
| `/admin/users/:studentId/progress` | 某生毕业进度（只读） | 3.3 |
| `/admin/error-cases` | 错误案例治理 | 3.5 |
| `/admin/forum` | 论坛治理 | 3.5 |
| `/admin/audit`（可选） | 操作审计 | 3.7 |

**与当前仓库路由对齐说明**：已实现 `frontend/src/router/index.ts` 中的 `/admin/dashboard`、`/admin/users`、`/admin/users/:studentId/progress`、`/admin/forum`；上表中 **`/admin/error-cases`、`/admin/audit` 尚未有对应路由与页面**，错误案例接口仍走既有 `backend/app/api/v1/ai.py` 的 `/api/ai/error-cases` 能力（管理员分支在服务层）。

### 4.3 组件复用

- **毕业进度**：抽取学生端「进度展示」为纯展示组件，props 传入 `studentId` 或 `userId` + `readonly`；管理端只负责拉取「目标用户」的接口数据。  
- **表格**：统一分页、空状态、错误提示，便于答辩录屏。

## 5. 后端 API 规划草案（后期对齐 OpenAPI）

以下为**规划用**路径命名，实现时可并入 `backend/app/api/v1` 并加 `Depends(require_admin)` 一类依赖。

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/admin/dashboard/summary` | 用户数等摘要（P0 从简） |
| GET | `/api/admin/users` | 分页列表 + 筛选 query |
| GET | `/api/admin/users/{student_id}` | 用户详情 |
| PATCH | `/api/admin/users/{student_id}` | 启用禁用等（P1，body 限制字段） |
| GET | `/api/admin/users/{student_id}/academic-info` | 教务视角完整学业信息，复用学生端展示结构 |
| GET | `/api/admin/users/{student_id}/graduation-progress` | 教务视角毕业进度（P0） |
| POST | `/api/admin/users/{student_id}/warnings` | 给指定学生发送学业预警（P1，下次登录弹窗且只弹一次） |
| GET | `/api/admin/users/{student_id}/schedule` | 课表只读（P1，若独立） |
| GET | `/api/admin/forum/topics` | 管理员查询论坛帖子列表 |
| DELETE | `/api/admin/forum/topics/{topic_id}` | 管理员隐藏/软删除帖子 |
| PATCH | `/api/admin/forum/topics/{topic_id}` | 管理员恢复展示或再次隐藏（body：`status` 为 `normal` / `deleted`） |

错误案例、论坛等可在现有路由上增加 `admin` 分支或统一到 `/api/admin/...`，以**少重复、易测试**为准。

## 6. 分期实现建议（供排期）

| 阶段 | 内容 | 验收标准 |
|------|------|----------|
| **M1** | 首个 admin 创建方式 + `GET /me` 驱动前端路由守卫 + 空壳 `/admin` 首页 | admin 登录可进工作台，学生被挡 |
| **M2** | `require_admin` 后端依赖 + 用户列表 + 用户详情 + 按学号查看毕业进度（只读）+ harness | 答辩演示「看见任意学号进度」，普通学生访问管理接口返回 403 |
| **M3** | 学业预警发送 + 论坛治理（隐藏/删除帖子，支持取消隐藏） | 学生下次登录可看到一次性预警弹窗，违规帖子可从学生端隐藏并可恢复 |
| **M4** | 错误案例管理 UI、课表只读 Tab、统计卡片 | 按时间选做 |
| **M5** | 审计、导出 | 加分项 |

## 7. 非目标与风险（写清避免 scope creep）

- **非目标**：教务批量改成绩、改培养方案规则、替学生提交表单（除非单独立项）。  
- **风险**：管理员账号泄露 → 强密码、仅内网部署演示、生产环境务必 HTTPS + 最小管理员数。  
- **风险**：进度接口未校验角色 → 越权；实现时必须服务端校验 `admin` 或本人。
- **风险**：前端只隐藏菜单但后端不鉴权 → 普通学生可手写请求访问管理接口；所有 `/api/admin/*` 必须后端强校验。
