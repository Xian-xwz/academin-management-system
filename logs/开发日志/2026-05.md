- **根本原因**: 全局 `thesis-generator` 只有通用流程，缺少当前学校参考论文的封面、摘要、目录、图表题注、致谢、参考文献和附录格式沉淀；原始模板是旧版 `.doc`，不能直接作为 Pandoc `--reference-doc` 使用。

### 前端静态资源旧缓存导致登录后空白 - 023

#### 问题发现
- **时间**: 2026-05-03 02:04
- **描述**: 用户登录页显示登录成功但无法进入系统，浏览器控制台报 `Failed to load module script: Expected a JavaScript-or-Wasm module script but the server responded with a MIME type of "text/html"`，多个动态导入 chunk 加载失败。
- **环境**: 公网站点 `https://academin-management-system.maplexian.cn`；Caddy 静态前端；Vite hash 资源；Chrome 浏览器。
- **影响**: 登录接口实际成功，但前端路由 chunk 无法执行，用户看到页面停留/空白，误以为登录失败。
- **优先级**: 高

#### 问题分析
- **根本原因**: 前端部署使用 `rsync --delete dist/` 删除旧 hash 资源，但浏览器仍缓存旧 `index.html` 或旧路由 chunk 引用；Caddy 的 SPA fallback 对任意缺失路径都返回 `/index.html`，导致 `/assets/旧文件.js` 缺失时返回 `200 text/html`，浏览器将 HTML 当作 JS module 加载而报错。
- **相关代码**: 服务器 `/etc/caddy/Caddyfile`、前端部署目录 `/var/www/academin-management-system`
- **数据流**: 浏览器旧入口文件 → 请求已删除的 `/assets/*.js` → Caddy fallback 到 `/index.html` → 返回 `text/html` → ES module MIME 校验失败。
- **复现步骤**: 部署新前端后不强刷页面，直接登录；查看控制台可见缺失 chunk 被返回 HTML。

#### 修复方案
- **时间**: 2026-05-03 02:07
- **解决方案**: 调整 Caddy：为 `/assets/*` 增加独立 `handle`，只走 `file_server`，缺失资源直接返回 404，不再 fallback 到 `index.html`；为 SPA 入口 fallback 增加 `Cache-Control: no-cache, no-store, must-revalidate`，避免部署后继续使用旧入口 HTML；保留 `/api/*` 反代和 `/dify-agent-tools.openapi.json` 专用路由。
- **修改文件**: 服务器 `/etc/caddy/Caddyfile`、本地 `logs/开发日志/2026-05.md`
- **代码变更**: Caddy 增加 `/assets/*` 静态资源规则和入口页 no-cache 头。
- **测试方案**: `caddy validate --config /etc/caddy/Caddyfile`；`systemctl reload caddy`；公网检查缺失旧资源、新资源、入口页和登录接口。
- **验证结果**: `caddy validate` 通过并 reload；缺失旧资源返回 `404 Not Found`；当前新资源返回 `200 text/javascript` 且带 immutable 缓存；首页返回 `Cache-Control: no-cache, no-store, must-revalidate`；公网 `/api/health` 返回成功；公网登录 `202211911001 / guangdong11` 返回 token 和用户信息。

#### 经验总结
- **经验教训**: SPA fallback 不应覆盖 `/assets/*`，否则缺失资源会伪装成 HTML 成功响应，前端报错不直观。
- **预防措施**: 后续所有 Vite/SPA 部署固定使用“入口 HTML 不缓存、hash assets 长缓存、缺失 assets 返回 404”的 Caddy 规则。
- **文档更新**: 已更新本日志。

### 知识卡片上传图片生成结果取错 URL - 024

#### 问题发现
- **时间**: 2026-05-03 02:36
- **描述**: 用户测试“上传图片生成知识卡片”时，Drawer 里生成完成后展示的是用户上传的原图预览，而不是 Dify 生成的知识卡片图。
- **环境**: 公网页面 `https://academin-management-system.maplexian.cn/knowledge-cards`；后端知识卡片 SSE；Dify 知识卡片应用。
- **影响**: 图片输入场景下用户无法看到真正生成的知识卡片，历史保存的 output 也可能落成输入图片。
- **优先级**: 高

#### 问题分析
- **根本原因**: 后端 `_persist_output_image()` 对 `raw_response` 递归查找任意字段名为 `url` 的值；当 Dify 工作流包含上传文件、中间“提取 url”节点或文件预览事件时，`events` 中会出现 `.../files/{id}/file-preview...` 输入图 URL。旧逻辑没有区分“输入图预览 URL”和“生成图 URL”，因此可能把输入图下载并保存为 output。
- **相关代码**: `backend/app/services/knowledge_card.py`
- **数据流**: 用户上传图片 → 后端上传到 Dify 得到 input file-preview URL → Dify 工作流生成卡片并在 `answer` 返回 `.../files/tools/...png` → 后端旧逻辑递归扫 `events/url` → 误抓 input file-preview → 保存成 `/api/knowledge-cards/{id}/files/output`。
- **复现步骤**: 上传图片创建知识卡片；等待 Dify 工作流完成；观察 Drawer 预览图与上传原图一致，未展示生成图。

#### 修复方案
- **时间**: 2026-05-03 02:45
- **解决方案**: 新增 `_extract_output_image_url()`，只从最终 `answer`、`outputs`、明确的 `image_url/output_image_url/generated_image_url/card_image_url` 和 `files` 中提取候选图片 URL；新增 `_choose_output_image_url()` 过滤 `/file-preview`、插件 icon 等非生成图，并优先选择 `/files/tools/` 生成图 URL；不再对整个 `raw_response/events` 递归匹配任意 `url`。
- **修改文件**: `backend/app/services/knowledge_card.py`、`logs/开发日志/2026-05.md`
- **代码变更**: 将原来的宽泛 `_find_first(..., {"url"})` 改为最终输出优先、输入图预览排除的 URL 选择规则。
- **测试方案**: 本地 `python -m py_compile backend/app/services/knowledge_card.py`；云端部署后重新上传图片生成知识卡片，确认预览为 Dify 生成图。
- **验证结果**: 本地 `python -m py_compile backend/app/services/knowledge_card.py` 通过；云端 `git reset --hard origin/main` 后编译通过并重启 `academic-backend`，`/api/health` 返回成功。使用新规则检查已有图片输入记录时，候选 URL 已正确选择 Dify 生成图 `.../files/tools/...png`，不再选择输入图 `.../file-preview...`；但该历史记录的 Dify 签名生成图 URL 已过期返回 403，无法可靠回补，需以后续新生成结果回归。

#### 经验总结
- **经验教训**: Dify 流式事件中会混入节点输入、插件图标、上传文件预览和最终输出，图片落盘必须明确候选优先级，不能递归抓任意 `url`。
- **预防措施**: 后续解析多媒体输出时优先读取最终输出字段，并维护排除列表（输入预览、插件 icon、控制台资源等）。
- **文档更新**: 已更新本日志。
