### OpenClaw 基础设施 - 001

#### 问题发现
- **时间**: 2026-05-02 02:37
- **描述**: OpenClaw 网关图片工具报 `Model does not support images: google/gemini-3-flash-preview`，并出现 `Unknown model` 与飞书 WebSocket `system busy` 重试；同时存在工具反复读取不存在路径的行为，需要通过规则文件收敛。
- **环境**: 阿里云 ECS Ubuntu，OpenClaw Gateway，飞书 WebSocket，Google Gemini Provider。
- **影响**: 图片分析不可用，重试会拖慢网关响应；错误读文件会浪费上下文与工具调用次数。
- **优先级**: 高

#### 问题分析
- **根本原因**: 服务器全局 `openclaw` 版本为 `2026.4.15`，已安装插件要求更高版本；默认模型列表未在服务环境中正确呈现图片能力，且未显式配置 `imageModel`；根目录与 yellow 子工作区缺少“读文件前先确认路径存在”的行为约束。
- **相关代码**: 服务器 `/root/openclaw/red-maple/openclaw.json`、`/root/openclaw/red-maple/AGENTS.md`、`/root/openclaw/red-maple/TOOLS.md`、`/root/openclaw/red-maple/projects/mail-agent/agents/yellow/AGENTS.md`、`/root/openclaw/red-maple/projects/mail-agent/agents/yellow/TOOLS.md`
- **数据流**: 飞书/网关消息 → OpenClaw Gateway → 模型注册表与图片工具 → Google Gemini API。
- **复现步骤**: 在旧版本网关中调用图片分析工具，观察 `gateway.log` 中 `Model does not support images`、`Unknown model` 或飞书 WS 重试日志。

#### 修复方案
- **时间**: 2026-05-02 02:47
- **解决方案**: 备份配置与 systemd unit；将全局 npm 包 `openclaw` 从 `2026.4.15` 升级到 `2026.4.29`；保留 `agents.defaults.heartbeat.every = "6h"`；显式配置 `imageModel` 为 `google/gemini-2.5-flash`，并添加 `google/gemini-3.1-flash-image-preview` 作为 fallback；在根目录和 yellow 子工作区补充 `AGENTS.md` 文件读取纪律与 `TOOLS.md` 工作区结构说明；重启 `openclaw-red-maple-gateway.service`。
- **修改文件**: 服务器 `/root/openclaw/red-maple/openclaw.json`、`/root/openclaw/red-maple/AGENTS.md`、`/root/openclaw/red-maple/TOOLS.md`、`/root/openclaw/red-maple/projects/mail-agent/agents/yellow/AGENTS.md`、`/root/openclaw/red-maple/projects/mail-agent/agents/yellow/TOOLS.md`、本地 `logs/开发日志/2026-05.md`
- **代码变更**: 无本地业务代码变更；服务器配置新增图片模型注册和工具行为规则。
- **测试方案**: `openclaw --version`、带 service 环境变量执行 `openclaw models list`、`systemctl restart/status openclaw-red-maple-gateway.service`、本机访问网关 HTTP 端点、检查飞书 WS 日志与独立 `feishu-event-ws` 进程。
- **验证结果**: `openclaw --version` 为 `2026.4.29`；`openclaw models list` 显示 `google/gemini-2.5-flash` 为 `text+image` 且标记 `image`；网关 `active (running)` 并输出 `gateway ready`；内置飞书 WebSocket 启动，独立 `feishu-event-ws` 进程不存在；HTTP 端点返回 200。

#### 经验总结
- **经验教训**: OpenClaw CLI 默认读取路径与 systemd 服务环境可能不同，验证模型列表时必须带 `OPENCLAW_CONFIG_PATH`、`OPENCLAW_WORKSPACE_DIR` 和 `OPENCLAW_STATE_DIR`。
- **预防措施**: 升级或修改模型配置前先备份 `openclaw.json`、行为规则和 systemd unit；图片工具使用独立 `imageModel`，不要依赖主模型能力推断；工具读文件前必须确认路径存在。
- **文档更新**: 已更新 `AGENTS.md` 与 `TOOLS.md` 的行为规则和工作区结构说明；本条日志记录服务器侧变更。

### OpenClaw 接入 - 002

#### 问题发现
- **时间**: 2026-05-02 02:51
- **描述**: 需要按 `openclaw_接入部署_c743d142.plan.md` 第 12 行任务新增 `/api/openclaw` 受控工具层，让 OpenClaw 只能通过 FastAPI 封装调用本人只读数据和 AI 问询，不能复用管理员账号、直连 Dify 或调用写接口。
- **环境**: Windows 11，本地 FastAPI 后端，计划部署到阿里云 ECS。
- **影响**: 若直接开放现有业务接口，OpenClaw 可能越权查询任意学生、调用管理员接口或触发写操作。
- **优先级**: 高

#### 问题分析
- **根本原因**: 现有接口面向浏览器 JWT 用户设计，缺少专门的服务令牌、学生访问白名单和工具调用审计表；仓库也尚未实现 OpenClaw 会话绑定学生的一次性上下文令牌。
- **相关代码**: `backend/app/core/config.py`、`backend/app/core/security.py`、`backend/app/api/v1/openclaw.py`、`backend/app/services/openclaw.py`、`backend/app/models/openclaw.py`
- **数据流**: OpenClaw → `/api/openclaw/*` → 服务令牌校验 → 学号白名单校验 → 复用只读 service / AI service → 写入 `openclaw_tool_audits`。
- **复现步骤**: 在未实现前，OpenClaw 只能选择直接调用普通 `/api/student`、`/api/ai` 或管理员/写接口，缺少统一边界。

#### 修复方案
- **时间**: 2026-05-02 03:05
- **解决方案**: 新增 `OPENCLAW_TOOL_TOKEN` 与 `OPENCLAW_ALLOWED_STUDENT_IDS` 配置；新增 `require_openclaw_client` 服务令牌鉴权；新增 `OpenClawToolAudit` 审计模型与 `openclaw_tool_service`；新增 `/api/openclaw/health`、学业信息、毕业进度、课表、时间规划事件和非流式 AI 问询接口；注册路由并更新环境变量示例、后端 README、模型说明和计划状态。
- **修改文件**: `.env.example`、`backend/README.md`、`backend/app/api/v1/openclaw.py`、`backend/app/api/v1/router.py`、`backend/app/core/config.py`、`backend/app/core/security.py`、`backend/app/models/openclaw.py`、`backend/app/models/__init__.py`、`backend/app/models/README.md`、`backend/app/services/openclaw.py`、`c:\Users\1\.cursor\plans\openclaw_接入部署_c743d142.plan.md`、`logs/开发日志/2026-05.md`
- **代码变更**: 第一版采用服务令牌 + 学号 allowlist；`OPENCLAW_ALLOWED_STUDENT_IDS=*` 仅建议演示使用；审计只记录工具名、学号、请求摘要、响应状态、耗时和错误摘要，不记录密钥或完整大段输入。

### 部署与基础设施 - 003

#### 问题发现
- **时间**: 2026-05-02 03:10
- **描述**: 项目需要先达到“给出 URL 即可公网访问网页”的程度；服务器代码已克隆但后端 systemd 未启用，Caddy 仍将 `academin-management-system.maplexian.cn` 反代到被 mail-agent 占用的 `8080`，数据库尚未建表和导入业务数据。
- **环境**: 阿里云 ECS Ubuntu，MySQL 8.0，Caddy，FastAPI，Vue 3 静态构建。
- **影响**: 用户无法通过公网域名访问毕业设计系统；后端接口和数据链路不可用。
- **优先级**: 高

#### 问题分析
- **根本原因**: 服务器 `.env` 缺少数据库连接、上传目录、CORS 和 OpenClaw 工具配置；`DATABASE_URL` 若使用带空格库名的百分号编码会被 MySQL 当作字面库名；前端尚未构建发布，Caddy 站点块也未按静态页面 + `/api` 分流配置。
- **相关代码**: 服务器 `/opt/academin-management-system/app/.env`、`/etc/systemd/system/academic-backend.service`、`/etc/caddy/Caddyfile`、本地 `README.md`
- **数据流**: 浏览器 → Caddy HTTPS → 静态前端 `/var/www/academin-management-system`；`/api/*` → FastAPI `127.0.0.1:8081` → MySQL `Academic Management System`。
- **复现步骤**: 访问 `https://academin-management-system.maplexian.cn` 或 `/api/health`，旧配置会反代到错误端口或后端未启动。

#### 修复方案
- **时间**: 2026-05-02 03:18
- **解决方案**: 使用 MySQL root socket 执行一次性建库与专用账号 SQL；服务器 `.env` 改用 `MYSQL_*` 字段处理带空格库名；运行 `init_schema.py`、`import_knowledge_json.py`、`seed_scenario_data.py` 和 `check_db.py`；新增并启动 `academic-backend.service`；前端执行 `npm install` 与 `npm run build`，发布到 `/var/www/academin-management-system`；Caddy 改为静态站点并将 `/api/*` 反代到 `127.0.0.1:8081`。
- **修改文件**: 服务器 `/opt/academin-management-system/app/.env`、服务器 `/etc/systemd/system/academic-backend.service`、服务器 `/etc/caddy/Caddyfile`、本地 `README.md`、`c:\Users\1\.cursor\plans\openclaw_接入部署_c743d142.plan.md`、`logs/开发日志/2026-05.md`
- **代码变更**: 无新增业务代码；服务器同步了本地 `/api/openclaw` 后端文件以保证表模型和路由与当前代码一致。
- **测试方案**: 本机后端 `/api/health`、公网首页、公网 `/api/health`、公网学生登录接口、数据库脚本输出和 systemd/Caddy 状态检查。
- **验证结果**: `https://academin-management-system.maplexian.cn/` 返回前端 HTML；`https://academin-management-system.maplexian.cn/api/health` 返回 `{"status":"ok","db":"ok"}`；学生 `202211911001` / `123456` 登录成功并返回 token；MySQL 已创建 17 张表，导入 48 个专业、3278 门课程、860 条实践环节和 18 个演示学生。

#### 经验总结
- **经验教训**: 数据库名包含空格时，不要在手写 `DATABASE_URL` 中使用百分号编码给 MySQL 驱动；改用配置层 `URL.create` 或避免空格库名更稳妥。
- **预防措施**: 部署文档中固定端口：后端 `8081`、Caddy `/api` 分流；正式展示前替换演示密码，保管服务器 `.env`。
- **文档更新**: 已在根 `README.md` 增加公网 URL、服务名、静态目录和演示账号说明；计划文件前两项已标记完成。

### 后端演示数据 - 004

#### 问题发现
- **时间**: 2026-05-02 13:47
- **描述**: 现有论坛 mock 动态只在 `seed_scenario_data.py` 中写死电科专业 3 个话题和 3 条评论，登录时的 mock 互动也主要围绕当前用户已有帖子，导致大部分专业和学生账号登录后论坛内容不够丰富。
- **环境**: Windows 11，本地 FastAPI 后端，计划同步到阿里云 ECS。
- **影响**: 云端演示时非电科学生登录后论坛动态不足，首页通知与论坛交流氛围不明显。
- **优先级**: 中

#### 问题分析
- **根本原因**: 登录 mock 活动器缺少跨专业生成策略，也未接入 LLM；多数学生初始没有自己的帖子，旧逻辑无法为其专业创建新动态。
- **相关代码**: `backend/app/services/dashboard.py`、`backend/app/core/config.py`
- **数据流**: 学生登录 → `auth_service.login` → `dashboard_service.generate_login_mock_activity` → 根据学生专业发布论坛帖子或评论 → 首页通知与论坛列表展示。
- **复现步骤**: 使用非电科演示账号登录，查看论坛列表和首页通知，内容主要集中在电科或无新增动态。

#### 修复方案
- **时间**: 2026-05-02 14:05
- **解决方案**: 将登录 mock 活动器改造为“mock 动态器”：所有学生登录后按其专业选择发布新帖或评论已有帖子；优先调用 DashScope 兼容接口，失败后调用 Gemini，再失败则使用本地模板兜底；新增 `MOCK_DYNAMIC_*`、`DASHSCOPE_*`、`GEMINI_*` 配置说明。
- **修改文件**: `backend/app/services/dashboard.py`、`backend/app/core/config.py`、`.env.example`、`backend/README.md`、`logs/开发日志/2026-05.md`
- **代码变更**: 新增专业维度的论坛动态生成逻辑；登录失败不受 LLM 影响；DashScope `Model.AccessDenied` 时自动回退到 Gemini 或模板。
- **测试方案**: `python -m compileall backend/app`、IDE linter、直接调用 mock LLM 生成函数进行 smoke test。
- **验证结果**: 编译通过；相关文件 linter 无报错；本地 smoke test 可走到 Gemini fallback 返回生成文本。

#### 经验总结
- **经验教训**: 演示数据生成不能只覆盖一个专业；登录链路接 LLM 时必须设置超时和兜底，避免影响核心登录。
- **预防措施**: 云端部署需在 `.env` 配置 `GEMINI_API_KEY` 或可用 DashScope 模型；若演示时不希望自动生成，可设置 `MOCK_DYNAMIC_ENABLED=false`。
- **文档更新**: 已更新 `.env.example`、`backend/README.md` 和本日志。
