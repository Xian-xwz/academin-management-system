### 部署与基础设施 - 002

#### 问题发现
- **时间**: 2026-04-28 16:47
- **描述**: 自托管 Dify 调整插件调试端口映射后，`docker-api-1` 反复重启；日志报 `ValidationError: PLUGIN_REMOTE_INSTALL_PORT Input should be a valid integer`，`input_value='127.0.0.1:5003'`。
- **环境**: 阿里云 ECS Ubuntu，Docker Compose 部署 Dify 1.13.3
- **影响**: API 无法启动，控制台与 nginx 上游返回 502。
- **优先级**: 高

#### 问题分析
- **根本原因**: `.env` 中 `EXPOSE_PLUGIN_DEBUGGING_PORT` 被写成 `127.0.0.1:5003`；Dify 后端将该变量解析为整数端口，不接受 `host:port` 形式。
- **相关代码**: 运行时环境变量校验（`DifyConfig` / Pydantic）；`docker/docker-compose.yaml` 中 `plugin_daemon` 的 `ports` 映射。
- **数据流**: `.env` → Compose 展开 → 容器内 `PLUGIN_REMOTE_INSTALL_PORT` 等 → API 进程启动校验失败。
- **复现步骤**: 将 `EXPOSE_PLUGIN_DEBUGGING_PORT=127.0.0.1:5003` 写入 `.env` 后执行 `docker compose up -d`，观察 `docker compose logs api`。

#### 修复方案
- **时间**: 2026-04-28 16:49
- **解决方案**: 将 `EXPOSE_PLUGIN_DEBUGGING_PORT` 恢复为纯端口 `5003`；在 `docker-compose.yaml` 中将插件调试端口映射改为 `127.0.0.1:${EXPOSE_PLUGIN_DEBUGGING_PORT:-5003}:5003`，使宿主机仅本机监听，满足安全收紧且不与整数校验冲突。
- **修改文件**: 服务器 `/opt/dify/docker/.env`、`/opt/dify/docker/docker-compose.yaml`（本地仓库未同步时以服务器为准）。
- **代码变更**: 端口映射由 `0.0.0.0:5003` 改为 `127.0.0.1:5003`；`.env` 端口变量仅保留数字。
- **测试方案**: `docker compose ps api` 为 healthy；`curl -I http://127.0.0.1:18080/install` 返回 200；`ss -tlnp` 确认 5003 仅监听 127.0.0.1。
- **验证结果**: API 健康；本机访问 Dify nginx 正常；若此前因 web 容器重建导致 nginx 缓存旧上游 IP，执行 `docker compose restart nginx` 后 502 消失。

#### 经验总结
- **经验教训**: Dify 的 `EXPOSE_*` 在部分变量上表示「宿主机映射侧端口」，语义与「仅写 bind 地址」不同，不能把 `ip:port` 写进要求整数的项。
- **预防措施**: 需要「仅本机暴露」时优先改 Compose `ports` 的 bind 地址，`.env` 保持官方示例类型。
- **文档更新**: 本记录；`plan/mineru知识库处理_812ac5d4.plan.md` 补充自托管访问基址与 DNS 说明。

### 部署与基础设施 - 004

#### 问题发现
- **时间**: 2026-04-28 18:08
- **描述**: 用户反馈 `https://dify.maplexian.cn` 打不开；外部页面资源能返回 200，但关键接口 `/console/api/setup` 返回 502。
- **环境**: 阿里云 ECS Ubuntu，Dify 1.13.3 Docker Compose，Caddy 反代 `https://dify.maplexian.cn`
- **影响**: Dify 安装页/控制台前端可以加载静态资源，但初始化接口失败，用户表现为页面打不开或卡住。
- **优先级**: 高

#### 问题分析
- **根本原因**: API 容器重建后 IP 已变化，Dify 内置 nginx 仍把 `/console/api/*` 转发到旧的 API 容器 IP，日志显示 `connect() failed (111: Connection refused) while connecting to upstream`。
- **相关代码**: 服务器 `/opt/dify/docker/docker-compose.yaml`、容器 `docker-nginx-1` 的运行时上游解析状态。
- **数据流**: 浏览器 → Caddy 443 → `127.0.0.1:18080` → Dify 内置 nginx → `api:5001`；故障发生在内置 nginx 到 API 容器这一段。
- **复现步骤**: 访问 `https://dify.maplexian.cn/install` 后查看浏览器接口请求，或在服务器执行对 `/console/api/setup` 的请求，返回 502。

#### 修复方案
- **时间**: 2026-04-28 18:10
- **解决方案**: 按既有记录中的最小恢复方案，仅重启 Dify 内置 nginx 容器刷新上游解析，不修改 Caddy、数据库、源码补丁和 `.env`。
- **修改文件**: 无；仅执行服务器运行时操作 `docker compose restart nginx`。
- **代码变更**: 无。
- **测试方案**: 验证 `https://dify.maplexian.cn/install`、`/console/api/setup`、`/console/api/system-features`。
- **验证结果**: 三个地址均返回 200；`/console/api/setup` 返回 `{"step":"finished","setup_at":"2026-04-28T09:08:19"}`，说明控制台 API 已恢复。

#### 经验总结
- **经验教训**: Dify 内置 nginx 对容器上游解析存在缓存，API/web 重建后即使 API healthy，也可能因旧 IP 导致 502。
- **预防措施**: 后续重建 API、worker 或 web 后，若控制台 API 502，优先检查 nginx error log 中的 upstream IP，并重启内置 nginx 刷新解析。

### 前端依赖与启动 - 005

#### 问题发现
- **时间**: 2026-04-29 17:18
- **描述**: 拉取 AI Studio 生成的前端工程后，执行 `npm run dev` 启动 Vite 失败，先报 `Cannot find module '../lightningcss.win32-x64-msvc.node'`，后续修复 Tailwind 后又报 `Failed to resolve entry for package "element-plus"`。
- **环境**: Windows 11，PowerShell，Node/npm，本地 Vue 3 + Vite 前端工程 `frontend/`
- **影响**: 前端开发服务器无法正常启动，无法预览页面。
- **优先级**: 高

#### 问题分析
- **根本原因**: 前端工程使用 Tailwind CSS v4 的 `@tailwindcss/vite` 插件时依赖 `lightningcss` 原生模块，Windows 平台二进制加载失败；同时 `element-plus@2.13.7` 安装结果中 `package.json` 指向的 `es/index.mjs` 缺失，导致 Vite 无法解析包入口。
- **相关代码**: `frontend/vite.config.ts`、`frontend/package.json`、`frontend/src/index.css`
- **数据流**: `npm install` → `node_modules` 依赖树 → Vite 读取 `vite.config.ts` → 加载 Tailwind / Element Plus → 启动失败。
- **复现步骤**: 在 `frontend/` 执行 `npm install` 后运行 `npm run dev`。

#### 修复方案
- **时间**: 2026-04-29 17:23
- **解决方案**: 将 Tailwind 从 v4 Vite 插件模式改为 Tailwind v3 + PostCSS 模式，移除 `@tailwindcss/vite` 对 `lightningcss` 的直接依赖；将 `element-plus` 降到稳定可解析版本 `2.9.11`，并同步安装 `@element-plus/icons-vue@2.3.1`。
- **修改文件**: `frontend/package.json`、`frontend/package-lock.json`、`frontend/vite.config.ts`、`frontend/src/index.css`、`frontend/tailwind.config.js`、`frontend/postcss.config.js`、`logs/开发日志/2026-04.md`
- **代码变更**: `vite.config.ts` 仅保留 Vue 插件；`src/index.css` 改为 `@tailwind base/components/utilities`；新增 Tailwind/PostCSS 配置文件；更新依赖版本。
- **测试方案**: 执行 `npm ls tailwindcss @tailwindcss/vite lightningcss --depth=0` 确认依赖树；运行 `npm run dev` 启动 Vite。
- **验证结果**: Vite 已成功启动；由于 3000 端口被其他进程占用，本次自动切换到 `http://localhost:3001/`。

#### 经验总结
- **经验教训**: AI Studio 生成的依赖版本可能包含平台原生包或较新的包入口变更，直接拉取后需要先做本地启动验证。
- **预防措施**: 前端工程优先使用稳定版本依赖；遇到原生模块加载失败时，优先考虑换成不依赖原生二进制的配置路径。
- **文档更新**: 已记录本次前端依赖修复过程。

- **时间**: 2026-04-30 23:08
- **描述**: 学业预警被实现为 `time_plan_events` 时间规划事件，学生首页待办、时间规划列表和日历都会展示该预警；实际产品口径只需要学生下次登录时弹出一次预警弹窗。
- **环境**: Windows 11，本地 Vite 前端、FastAPI 后端、MySQL。
- **影响**: 预警被错误归类为可编辑/可完成的日程事件，增加学生端噪音，也不符合“一次性提醒”的业务语义。
- **优先级**: 高

#### 问题分析
- **根本原因**: 之前为快速闭环复用时间规划事件作为预警载体，但学业预警属于管理员下发的只读提醒，不应参与待办、日历和时间规划 CRUD。
- **相关代码**: `backend/app/models/academic_warning.py`、`backend/app/services/admin.py`、`backend/app/services/auth.py`、`backend/app/schemas/auth.py`、`frontend/src/views/Login/index.vue`、`frontend/src/views/Dashboard/index.vue`、`frontend/src/views/TimePlan/index.vue`、`frontend/src/components/Calendar/index.vue`
- **数据流**: 管理员发送预警 → 写入 `academic_warnings` 且 `shown_at` 为空 → 学生登录 → `/api/auth/login` 返回 `pendingAcademicWarnings` → 后端写入 `shown_at` → 前端弹窗展示一次。
- **复现步骤**: 管理员在用户列表发送学业预警后，学生登录可在待办/时间规划看到“学业预警”事件，且不是一次性弹窗。

#### 修复方案
- **时间**: 2026-04-30 23:45
- **解决方案**: 新增 `AcademicWarning` 独立模型；管理员发送预警改为写入 `academic_warnings`；登录接口查询未展示预警并立即标记 `shown_at`；登录页收到 `pendingAcademicWarnings` 后弹窗展示；移除首页待办、时间规划类型下拉和日历颜色映射中的“学业预警”事件入口。
- **修改文件**: `backend/app/models/academic_warning.py`、`backend/app/models/__init__.py`、`backend/app/models/README.md`、`backend/app/schemas/auth.py`、`backend/app/schemas/admin.py`、`backend/app/services/auth.py`、`backend/app/services/admin.py`、`backend/scripts/harness_admin_api.py`、`frontend/src/views/Login/index.vue`、`frontend/src/views/Admin/Users/index.vue`、`frontend/src/views/Dashboard/index.vue`、`frontend/src/views/TimePlan/index.vue`、`frontend/src/components/Calendar/index.vue`、`backend/README.md`、`frontend/README.md`、`plan/教务工作台_管理员功能规划.md`、`logs/开发日志/2026-04.md`
- **代码变更**: `POST /api/admin/users/{student_id}/warnings` 返回 `warningId`；`POST /api/auth/login` 返回 `pendingAcademicWarnings` 并完成一次性标记；学生端不再把预警作为时间规划事件处理。
- **测试方案**: 运行 `init_schema.py` 创建新增表；后端 Python 编译；管理员 harness 验证预警首次登录返回、第二次登录为空；前端 TypeScript 检查与 IDE linter。
- **验证结果**: `conda run -n ams python scripts/init_schema.py` 创建 `academic_warnings` 表；后端 `py_compile` 通过；`conda run -n ams python scripts/harness_admin_api.py` 通过，输出含 `student_login=200` 与 `student_warning_once=200`；`npm run lint`（前端 `tsc --noEmit`）通过；相关变更文件 IDE linter 无报错。

#### 经验总结
- **经验教训**: 临时复用已有表虽然能快速演示，但要核对业务语义；只读一次性提醒不应落到可编辑事件表。
- **预防措施**: 后续涉及“通知/提醒/预警”先明确展示位置、是否可重复展示、是否可编辑和是否需已读态，再选择数据模型。
- **文档更新**: 已更新 `backend/README.md`、`backend/app/models/README.md`、`frontend/README.md` 与 `plan/教务工作台_管理员功能规划.md`。
