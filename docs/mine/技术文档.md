#### 1.2.2 后端技术栈
- **开发语言**: Python 3.9+
- **Web框架**: FastAPI
- **ORM框架**: SQLAlchemy
- **API文档**: Swagger / OpenAPI（FastAPI 自动生成）

#### 1.2.3 数据库
- **关系型数据库**: MySQL 8.0+（部署于轻量服务器）
- **向量数据库**: Chroma（嵌入式，Python 库，无需独立部署服务）
- **缓存**: 应用层内存缓存（毕设规模暂不引入 Redis，降低服务器内存压力）

#### 1.2.4 AI平台
- **AI问询平台**: 云端 Dify Chatflow + Dify 知识库（负责培养方案、毕业要求、选课规则等自然语言问答）
- **业务计算层**: FastAPI 代码实现毕业进度、缺口学分、课表、日历等确定性业务逻辑，避免将简单 CRUD 强行放入工作流
- **自建知识库方案**: LLM Wiki 离线编译 + Chroma 嵌入式向量检索（作为本地 RAG 实现与论文技术亮点，详见第 1.4 节）
- **扩展智能体**: OpenClaw 作为后续个人学习助手扩展模块，可调用 Dify Chatflow 或 FastAPI 接口，不作为系统主链路依赖
- **大模型**: 通过 Dify 或直接 LLM API 接入云端模型（DeepSeek / 通义千问 / GPT 等国内外模型均支持）

#### 1.2.5 文件存储
- **附件存储**: 服务器本地目录（通过 Nginx 对外提供静态文件访问）
- **对象存储**: 阿里云OSS / 腾讯云COS（可选，超出本地容量时启用）

#### 1.2.6 部署与运维
- **反向代理**: Nginx（利用服务器自带域名，统一代理前端静态文件与后端 API）
- **进程管理**: Gunicorn + Uvicorn（FastAPI 生产部署）
- **容器化**: Docker（可选，主要用于本地开发环境一致性）

## 1.4 两阶段知识构建方案

### 1.4.1 设计背景

人才培养方案 PDF 包含大量跨页表格与多级标题，直接对原始文本分块进行向量化（传统 RAG）存在以下问题：

- 表格跨页后分块语义断裂（如学分要求被切割为两段）
- 同一专业信息分散在多处，检索难以一次性聚合
- 数字型字段（如 32 学分、28 学分）在向量空间中语义区分度低

### 1.4.2 方案设计：离线编译 + 运行时检索

```
【离线阶段（一次性，更新人才培养方案时重跑）】

广东海洋大学本科专业人才培养方案.pdf
            │
            ▼ markitdown 解析
        raw/培养方案.md（原始文本）
            │
            ▼ LLM 读原文，编译为结构化知识页（参考 LLM Wiki 模式）
    wiki/
    ├── majors/计算机科学与技术.md    ← 专业页：毕业学分、各类要求等
    ├── majors/软件工程.md
    ├── concepts/毕业要求.md          ← 概念页：总学分/必修/选修等
    ├── concepts/课程分类.md          ← 各课程类别说明
    └── summaries/培养方案总览.md
            │
            ▼ Chroma 嵌入式向量化（一次性写入）
        chroma_db/（本地目录，轻量，无需独立服务）

【运行时】

学生提问 → FastAPI → Dify Chatflow / Dify 知识库 → 返回问询回答
                   │
                   ├── 需要确定性计算：FastAPI 查询 MySQL 并计算毕业进度
                   └── 自建 RAG 方案（可选）：Chroma 检索 wiki 页面 → LLM 生成回答
```

### 1.4.3 技术选型说明

| 组件 | 选型 | 选型原因 |
|------|------|----------|
| 文档解析 | markitdown | 支持 PDF/Word/PPT 等多格式转 Markdown |
| LLM Wiki 编译 | 参考 tonbistudio/llm-wiki 模板 | 将非结构化 PDF 编译为有组织的知识页面 |
| 向量库 | Chroma（嵌入式） | Python 原生调用，无独立服务，内存占用约 100-200MB |
| 向量模型 | 通过 LLM API 调用 text-embedding 类模型 | 无需本地 GPU |
| 检索方式 | 语义向量检索 + 关键词过滤 | 兼顾召回率与精确度 |

> 说明：毕设主问询链路优先采用 Dify Chatflow + Dify 知识库，保证快速落地；LLM Wiki + Chroma 作为自建 RAG 方案，用于提升技术完整性，也可作为 Dify 知识库效果不稳定时的本地兜底方案。

### 1.4.4 Chroma 嵌入式调用示例

```python
import chromadb
from chromadb.utils import embedding_functions

# 初始化本地嵌入式 Chroma（不需要独立服务进程）
client = chromadb.PersistentClient(path="./chroma_db")

# 使用 OpenAI 兼容的 Embedding（国产模型如 DashScope 同样支持）
embedding_fn = embedding_functions.OpenAIEmbeddingFunction(
    api_key=os.getenv("EMBEDDING_API_KEY"),
    api_base=os.getenv("EMBEDDING_API_BASE"),   # 可指向 DashScope 等兼容端点
    model_name="text-embedding-v3"
)

# 获取或创建知识库集合
collection = client.get_or_create_collection(
    name="academic_knowledge",
    embedding_function=embedding_fn
)

def search_knowledge(query: str, major: str = None, top_k: int = 5):
    """
    语义检索知识库，支持按专业过滤
    """
    where_filter = {"major": major} if major else None
    results = collection.query(
        query_texts=[query],
        n_results=top_k,
        where=where_filter
    )
    return results["documents"][0]

def ingest_wiki_pages(wiki_dir: str):
    """
    将 LLM Wiki 编译后的 Markdown 页面批量写入 Chroma
    """
    import os
    from pathlib import Path

    docs, ids, metas = [], [], []
    for md_file in Path(wiki_dir).rglob("*.md"):
        content = md_file.read_text(encoding="utf-8")
        # 按 H2 标题分段，保留语义完整性
        chunks = split_by_heading(content)
        for i, chunk in enumerate(chunks):
            docs.append(chunk)
            ids.append(f"{md_file.stem}_{i}")
            metas.append({"source": str(md_file), "major": extract_major(md_file)})

    collection.add(documents=docs, ids=ids, metadatas=metas)
```

### 2.2 向量数据库设计

#### 2.2.1 Chroma 集合设计
使用 Chroma 嵌入式向量库存储 LLM Wiki 编译后的知识页面（非原始 PDF chunk）：

**集合名称**: `academic_knowledge`

**字段定义**:
- `id`: 文档块唯一标识（字符串，格式：`页面名_块序号`）
- `documents`: 知识页面文本块内容（Markdown 段落）
- `embeddings`: 向量嵌入（float array，维度取决于 Embedding 模型，如 1536）
- `metadatas`: 元数据（JSON）
  - `major_code`: 专业代码（用于按专业过滤检索）
  - `page_type`: 页面类型（`major` / `concept` / `summary`）
  - `source`: 来源 wiki 文件路径
  - `section`: 所属章节标题

## 4. AI平台集成方案

本系统采用「FastAPI 业务计算 + Dify Chatflow 知识库问答」的集成方式。毕业进度、缺口学分、课表、日历等确定性业务逻辑由代码实现；Dify 负责自然语言理解、知识库检索和回答组织。这样既能利用 Dify 的低代码 AI 问询能力，也避免把简单 CRUD 和可测试的业务规则放进工作流中。

### 4.1 Dify Chatflow 与知识库配置

#### 4.1.1 知识库配置
1. 在 Dify 平台创建「学业培养方案知识库」
2. 上传人才培养方案 PDF，或上传 LLM Wiki 编译后的 Markdown 知识页
3. 配置文档分段策略和向量化模型
4. 设置知识库检索参数（Top K、相似度阈值等）

#### 4.1.2 Chatflow 配置
1. 创建 Chatflow 类型应用，定位为「学业问询助手」
2. 系统提示词限定回答范围：培养方案、毕业要求、课程分类、选课规则等
3. 接收后端传入的上下文变量，例如：
   - `student_id`: 学号
   - `major`: 专业名称
   - `grade`: 年级
   - `graduation_progress`: 后端代码计算出的毕业进度摘要
4. Chatflow 根据用户问题检索 Dify 知识库，并结合后端上下文生成自然语言回答

#### 4.1.3 FastAPI 与 Dify 的职责边界

| 功能 | 实现位置 | 原因 |
|------|----------|------|
| 学号校验、登录鉴权 | FastAPI | 规则确定，便于测试 |
| 已修课程、学分统计 | FastAPI + MySQL | 结构化数据查询，代码实现更可靠 |
| 毕业进度计算 | FastAPI | 涉及明确公式和边界条件，不能依赖模型自由生成 |
| 培养方案解释 | Dify Chatflow + 知识库 | 适合自然语言问答和文档检索 |
| 选课建议话术 | FastAPI 先算结果，Dify 再组织表达 | 兼顾准确性和可读性 |
| 日历/课表增删改 | FastAPI | 标准 CRUD，直接写代码更简单 |
| OpenClaw 个人助手 | 后续扩展 | 可调用 FastAPI 或 Dify，不作为主链路依赖 |

### 4.2 Dify Chatflow API 调用封装

```python
import requests

class DifyClient:
    def __init__(self, api_key, base_url):
        self.api_key = api_key
        self.base_url = base_url
        self.headers = {
            "Authorization": f"Bearer {api_key}",
            "Content-Type": "application/json"
        }

    def chat(self, query, user_id, inputs=None, conversation_id=None):
        """
        调用 Dify Chatflow API。
        inputs 用于传入后端计算好的学生上下文，避免模型自行猜测。
        """
        url = f"{self.base_url}/v1/chat-messages"
        data = {
            "inputs": inputs or {},
            "query": query,
            "user": user_id,
            "conversation_id": conversation_id,
            "response_mode": "blocking"
        }
        response = requests.post(url, json=data, headers=self.headers, timeout=30)
        response.raise_for_status()
        return response.json()
```

### 4.3 AI 问询服务（FastAPI 后端集成）

```python
import os
from dify_client import DifyClient
from database import get_student_info
from academic_service import calculate_graduation_progress

class AIService:
    def __init__(self):
        self.dify_client = DifyClient(
            api_key=os.getenv("DIFY_API_KEY"),
            base_url=os.getenv("DIFY_BASE_URL")
        )

    async def handle_question(self, question: str, student_id: str, conversation_id: str = None):
        """
        AI 问询主入口：
        1. 后端先查询学生信息并计算毕业进度；
        2. 将结构化结果作为上下文传给 Dify；
        3. Dify 负责结合知识库生成自然语言回答。
        """
        student_info = await get_student_info(student_id)
        progress = await calculate_graduation_progress(student_id)

        inputs = {
            "student_id": student_id,
            "major": student_info.get("major"),
            "grade": student_info.get("grade"),
            "graduation_progress": progress.to_summary_text()
        }

        return self.dify_client.chat(
            query=question,
            user_id=student_id,
            inputs=inputs,
            conversation_id=conversation_id
        )
```

## 15. 技术难点与解决方案

### 15.1 Dify Chatflow 与业务系统集成
**难点**: 如何让 Dify 负责自然语言问答，同时保证毕业进度等关键结果由后端代码准确计算
**解决方案**: 
- Dify 只负责 Chatflow、知识库检索和自然语言表达
- FastAPI 负责学号校验、毕业进度计算、课表/日历等确定性逻辑
- 后端将学生专业、年级、毕业进度摘要作为 `inputs` 传入 Dify，减少模型幻觉

### 15.2 PDF知识库构建（两阶段方案）
**难点**: 人才培养方案 PDF 含大量跨页表格，传统分块 RAG 语义易断裂
**解决方案**:
- 离线阶段：使用 markitdown 解析 PDF，再调用 LLM 将原始文本编译为结构化 Markdown 知识页（LLM Wiki 模式）
- 运行时：Chroma 嵌入式向量库对知识页进行语义检索，避免对原始乱序 chunk 检索
- 检索结合关键词过滤（按专业 metadata 筛选），提升精确度

### 15.3 跨端数据同步
**难点**: 暂不涉及（已取消小程序端，仅 Web 端）
**方案**: Web 端响应式设计，兼顾桌面和手机浏览器

### 15.4 AI问答准确率提升
**难点**: 提高AI回答的准确性和相关性
**解决方案**:
- 优化知识库质量
- 改进系统提示词
- 构建错题集持续优化
- 使用RAG（检索增强生成）技术

## 17. 总结

本文档详细描述了基于AI的学业管理系统的技术架构、数据库设计、API设计、Dify平台集成方案、部署方案等关键技术内容。系统采用前后端分离架构，由 FastAPI 实现确定性业务逻辑，通过 Dify Chatflow 与知识库实现学业问询能力。

### 17.1 技术亮点
1. **职责清晰**: FastAPI 负责毕业进度、学分统计等确定性计算，Dify Chatflow 负责知识库问答和语言表达
2. **两阶段知识构建**: LLM Wiki 离线编译培养方案 PDF 为结构化知识页，可结合 Chroma 进行本地语义检索，解决传统分块 RAG 的语义断裂问题
3. **轻量部署**: 业务库使用服务器已有 MySQL，Chroma 嵌入式向量库无需独立服务，适配 4GB 内存轻量服务器
4. **可扩展智能体**: OpenClaw 可作为后续个人学习助手扩展，调用 FastAPI 与 Dify，不影响主系统稳定性

### 17.2 后续优化方向
1. 引入更多AI能力（如课程推荐算法）
2. 支持更多数据源（如成绩分析、学习报告）
3. 优化用户体验（如个性化推荐）
4. 扩展功能（如学习社区、在线课程等）
