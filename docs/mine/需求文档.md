# 基于 AI 的学业管理系统 — 需求说明（由当前前端实现逆向整理）

> **说明**：本文档根据仓库内 `frontend/` 实际页面、路由、状态与 `src/api/modules` 的 mock 行为整理，用于与你心中的「正式需求」对照审核。凡标注 **mock** 的均为演示数据或未接后端；与真实教务数据、权限模型不一致之处，以你审核后的修订为准。

---

## 1. 系统定位与范围

- **定位**：面向在校学生的 Web 学业助手，提供学号维度的学业信息展示、课表与备注、时间规划、论坛交流、以及基于大模型的学业问询（当前为 mock 问答）。
- **范围（当前前端已实现页面）**：登录、注册、工作台首页、学号查询、AI 问询、论坛（列表/发帖/详情）、课表、时间规划、个人主页（含主题色本地设置）。
- **明确未实现或仅占位**：教师端独立登录、真实文件上传与存储、消息通知与待办的后端持久化、深色模式、修改密码真实流程、流式 SSE 输出等。

## 2. 技术栈与全局约定（与实现对齐）

| 项 | 当前实现 |
|----|----------|
| 框架 | Vue 3（`<script setup>`）、Vue Router 4、Pinia |
| UI | Element Plus、部分 Tailwind 工具类 |
| HTTP | Axios 封装于 `src/api/index.ts`，`baseURL` 为 `import.meta.env.VITE_API_BASE` 或默认 `/api` |
| 鉴权 | 请求头 `Authorization: Bearer {token}`；Token 存 `localStorage` 键 `Student-System-Token`；用户信息键 `Student-System-User` |
| 路由守卫 | 除 `/login`、`/register` 外均需 Token；无 Token 跳转 `/login?redirect={目标路径}`；已登录访问登录/注册页会重定向到 `/` |
| 401 | 响应拦截器清除 Token、提示过期并跳转登录 |

## 4. 用户与认证模块

### 4.1 登录（`/login`）

- **字段**：学号/用户名 `username`（必填）、密码 `password`（必填）。
- **交互**：支持回车提交；提交前 Element Plus 表单校验；按钮 loading。
- **成功**：调用 `login()`（当前 **mock**），将返回的 `token` 写入 Pinia + localStorage，将 `userInfo` 写入 Pinia + localStorage，提示成功，跳转 `/dashboard`。
- **失败**：`ElMessage` 显示错误信息。

**mock 返回形态**（`src/api/modules/auth.ts`）：`{ token, userInfo: { name: 表单用户名或「测试用户」 } }`。  
→ **与后端对齐时注意**：登录响应应补充 `student_id`、`major`、`role` 等，否则 AI 问询里会用占位学号（见第 7 节）。

### 4.2 注册（`/register`）

- **字段**：学号 `studentId`、用户名 `username`、邮箱 `email`（邮箱格式）、密码、确认密码（须与密码一致）。
- **成功**：提示「注册成功，请登录」，跳转 `/login`。
- **当前**：`register()` 为 **mock**，无真实服务端校验。

### 4.3 退出登录

- 清除 Token 与 `userInfo`，跳转 `/login`。

- **快捷导航**：卡片跳转学号查询、AI 问询、论坛、课表、时间规划。

## 6. 学号查询模块（`/student-query`）

### 6.1 业务目标

用户输入学号，展示该生的基本信息、毕业要求进度、课程分类统计、已修课程列表、还需修读建议。

### 6.2 前端校验

- 学号非空。
- 格式：`^[A-Za-z0-9]{8,20}$`（见 `src/utils/validate.ts`）。与输入框提示「8–20 位」一致。

### 6.3 查询与展示

- 点击「查询」或回车：loading，调用 `getAcademicInfo(studentId)`（**mock**）。
- 有结果：使用 `AcademicInfoCard` 展示。
- 无结果：若已发起过查询且无数据，显示 `el-empty` 提示未查到。

### 6.4 当前 mock 数据结构（后端应对齐或映射）

接口层返回对象（字段名与组件绑定一致）：

```text
{
  baseInfo: { name, studentId, major, grade },
  graduationReq: { total, earned, compulsory, elective, practice },
  courses: [ { name, category, credit, score, term }, ... ],
  statistics: [ { category, earned, total }, ... ],
  suggestions: [ { category, gap, desc }, ... ]
}
```

**展示规则**：

- 毕业总进度条百分比：`round(earned / total * 100)`。
- 分类统计：每类 `earned/total` 与进度条；`total===0` 时进度为 0；`earned >= total` 时进度条 success。
- 建议区：按 `suggestions` 渲染告警，`gap` 为缺口学分。

## 7. AI 问询模块（`/ai-chat`）

### 7.1 页面结构

- 顶栏：会话标题下拉（「开启新会话」+ **mock** 历史会话列表）、展示当前 `conversation_id` 后缀、`新建会话` 按钮。
- 主体：`ChatWindow` 聊天区。
- **当前写死**：`ChatWindow` 传入 `current-major="计算机科学与技术"`（与组件内默认 `电子科学与技术` 不一致，属实现细节，后端应对齐「专业来源」：登录用户 / 个人档案 / 可配置项）。

### 7.2 会话与消息（前端状态）

- `Pinia`：`conversation_id`、mock `historySessions`。
- 新建会话：清空 `conversation_id`、清空消息并重新插入欢迎语。
- 切换历史会话：设置 `conversation_id`，`loadHistorySession` 用 **写死两条消息** 模拟历史（非真实拉取）。

### 7.3 聊天交互（`ChatWindow`）

- 首屏自动插入助手欢迎语（培养方案、进度、选课等说明）。
- 用户发送：文本 + 可选附件（选择文件、拖拽文件）；附件名会拼进用户消息前缀 `[📎 附件: ...]`；发送后清空附件列表。
- 发送条件：有文本或附件；loading 中不可重复发送。
- Enter：仅 **不含换行** 的 Enter 触发发送（`@keydown.enter.exact.prevent`）。
- 助手消息可展示：`intent` 中文标签、`sources` 折叠列表（`documentName/title`、`score` 显示为百分比、`content` 摘要）。
- 免责声明：输入区下方固定提示「内容由 AI 生成…」。

### 7.4 请求载荷（`SendChatRequest`，供后端契约参考）

调用 `sendMessage` 时组装：

| 字段 | 来源/说明 |
|------|-----------|
| `query` | 用户文本（含附件前缀） |
| `conversation_id` | 已有会话则带上 |
| `student_id` | `userStore.userInfo.studentId`，若无则用 **`mock_student_001`** |
| `major_name` | 来自 `ChatWindow` 的 `currentMajor` prop |
| `inputs` | `{ student_id, major_name }` |

### 7.5 意图推断（前端 `inferIntent`，mock）

根据 `query` 关键词归类：

- `graduation_requirements`：毕业条件/毕业要求/要修多少分
- `graduation_audit`：能不能毕业/还差多少/学分够不够
- `course_advice`：选课/下学期修什么/课程建议
- 否则 `general_qa`

### 7.6 当前 mock 回复逻辑摘要（`sendMessage`）

- 返回 `conversation_id`（无则生成 `conv-mock-时间戳`）。
- `graduation_audit`：`need_personal_data=true`，回答提示需个人数据。
- 问题含「电科」或「电子科学与技术」：返回 **写死** 的电子专业学分结构说明，并带 `sources`。
- `graduation_requirements`：通用说明 + mock `sources`。
- `course_advice`：通用选课建议。
- 其他：`general_qa` 通用说明。

**后端对齐建议**：意图最终以服务端或 Dify 为准；前端推断仅作占位。

## 8. 论坛模块

### 8.1 列表页（`/forum`）

- **标题**：专业学习资料论坛；「发布新话题」跳转发帖页。
- **筛选**：关键词 `searchQuery`（占位提示：标题/内容/标签）；专业下拉「全部 / 计算机 / 软件工程」；「查询」拉取列表。
- **请求参数**（调用 `getTopics`）：`{ q, major, sort: 'latest' }`（当前列表写死 `latest`，与 mock 内支持的 `hot` 未完全暴露到 UI）。
- **列表展示**：朋友圈式 feed；点击卡片展开/收起全文；展示标签、附件（展开后）、浏览数、点赞、评论数。
- **点赞**：前端本地 `likes++`，提示成功（**无持久化**）。
- **评论**：每条帖子底部快速输入框，回车或发送调用 `addComment`，成功后 **本地 push** 一条评论，`author` 为 `userStore.userInfo.name` 或「当前测试用户」。

### 8.2 发帖页（`/forum/topics/create`）

- **必填**：标题、正文、所属专业（下拉固定几个专业）。
- **标签**：多选 + 可创建自定义标签（回车添加）。
- **附件**：`FileUpload` 组件，**明确为 Mock**，`auto-upload=false`，仅本地文件列表展示，提示不超过 50MB。
- **提交**：`createTopic(form)` mock，成功后回 `/forum`。

### 8.3 详情页（`/forum/topics/:id`）

- 展示标题、作者、专业、时间、标签、正文、附件列表（下载按钮为 **演示提示**）。
- 发表评论：校验非空，调用 `addComment`，成功后 **本地 unshift** 评论。

### 8.4 mock 数据形态（列表/详情共用字段）

帖子：`id, title, summary?, content, author, major, tags[], views, likes, replies, createTime, attachments?, comments[]`  
评论：`id, author, content, createTime, replies[]`  
回复：`id, author, content, createTime`

## 9. 课表模块（`/schedule`）

### 9.1 功能

- 学期下拉（写死两个学期选项）、周次 1–20、搜索框（课程名/教师/地点，不区分大小写包含匹配）。
- 数据：`getSchedule({ term, week })` mock 返回 `{ term, week, courses }`。
- **课表网格**：周一–周日 × 五个大节次（每大节对应两小节，内部用 `sections` 数组匹配）；单元格展示课程名、地点、周次、教师；有 `note` 时显示角标。
- **点击有课单元格**：弹出备注编辑；可保存或清空（清空即保存空字符串）；保存调用 `updateCourseNote(id, note)` mock 后 **更新本地数组**。

### 9.2 课程项字段（前端使用）

`id, name, teacher, location, day(1-7), sections(number[]), weeks, note`

## 10. 时间规划模块（`/time-plan`）

### 10.1 视图

- **列表视图**：表格展示事件标题、类型、起止时间、状态、编辑/删除。
- **日历视图**：`el-calendar` 按日聚合 `startTime` 前缀为 `YYYY-MM-DD` 的事件；点击某日打开新增弹窗并预填该日 08:00–09:00。

### 10.2 操作

- **从课表同步**：`syncFromSchedule()` mock，提示同步条数，刷新列表。
- **新增/编辑**：弹窗表单字段：标题（必填）、类型（必填：课程/考试/作业/个人）、时间范围（必填，`datetimerange`，`value-format="YYYY-MM-DD HH:mm"`）、地点、描述。提交时拆成 `startTime`、`endTime`。
- **删除**：二次确认后删除。

### 10.3 事件字段（mock 与表单）

`id, title, type, startTime, endTime, location, desc, status`（新增默认 `待开始`）

## 11. 个人主页（`/profile`）

- **展示**：头像首字、`userStore.userInfo.name`、根据 `userInfo.role === 'admin'` 显示「系统管理员」否则「学生」（当前登录 mock 未设置 `role`，一般为学生展示）。
- **学业档案**：姓名来自 store；学号、学院、专业、班级、入学年份等 **当前为写死占位**，与登录用户信息未联动。
- **系统外观**：主题强调色选择 + 取色器，写入 `localStorage` 键 `theme-color`，并在 `App.vue` 挂载时设置 `--el-color-primary`；深色模式开关禁用。

## 13. 与后端对接时的契约建议（供你审核）

1. **统一响应包**：建议约定 `{ code, message, data }`，前端 Axios 拦截器再适配（当前直接 `return response.data`）。
2. **登录响应**：至少 `token` + `userInfo`：`name`、`studentId`、`majorCode`、`majorName`、`grade`、`role`。
3. **`GET /api/student/{student_id}/academic-info`**：响应体与第 6.4 节字段一致或可映射。
4. **`POST /api/ai/chat`**：请求体支持第 7.4 节字段；响应体含 `answer`、`conversation_id`、`sources`、`intent`、`need_personal_data`。
5. **论坛/课表/时间规划**：按上文字段设计 REST 资源即可；文件上传需改为真实 `multipart` 与存储 URL。

## 14. 已知与正式需求可能冲突的点（请重点审核）

1. **AI 页专业写死** vs **用户实际专业**：需统一来源（用户档案 / 学号查询结果）。
2. **登录 mock 无 `studentId`**：AI 请求回落到 `mock_student_001`，与真实身份不一致。
3. **个人主页学业档案写死**：与认证信息、学号查询应否打通。
4. **课表 `getSchedule` 未使用周次参数**：mock 忽略 `week`，后端若按周返回需约定。
5. **论坛列表 sort 仅传 `latest`**：与「最热」类需求是否一致。
6. **论坛点赞仅前端自增**：需后端幂等与鉴权。
