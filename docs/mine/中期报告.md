基于AI的学业管理系统 — 中期报告

课题名称：基于AI大模型的智能学业管理系统（含Dify智能体，Web网页端实现）

说明：本报告对应学校中期检查要求，分三部分写：已做工作与进展、目前困难、后面安排。表述尽量贴近实际进度，不写超出能力范围的承诺，答辩时可逐项对照。

---

一、简述开题以来所做的具体工作和取得的进展或成果

开题以来，本人围绕开题报告确定的研究目标与技术路线，主要完成了基础准备与核心功能的前期开发工作。下面按模块说明具体工作与目前能展示到什么程度，避免写得太满，后续结题时以实际完成情况为准。

1. 文档与方案梳理

根据毕业设计要求，本人整理并编写了《需求文档》《技术文档》《前端指南》等材料。

需求文档里把系统要做什么写清楚，包括学号查询（毕业要求、已选课程、课程分类等）、AI问询助手、学习资料论坛、课表带备注、时间规划日历等，并标了功能编号，方便写论文时对照。

技术文档里写了打算用的技术选型，例如前后端分离、FastAPI、MySQL、Dify 等，以及接口设计思路，以后改接口也有据可查。

前端指南里把每个功能对应到网页上的哪些页面，方便自己开发时不乱。这些文档主要是给自己定规矩、给老师看进度用的，后面如果实现有调整，会同步改文档。

2. 基础环境与数据准备

开发环境方面，在本地安装了常用工具，用 Git 做版本管理，需要时用 Docker 起数据库等，具体视本机配置而定，以能跑通为准。

数据库方面，按技术文档里的思路设计了用户、专业、课程、选课、课表、时间规划、论坛等核心表，并写了部分建表或初始化脚本。目前表里以测试数据为主，真实教务数据若拿不到，毕业设计阶段会以「模拟数据 + 少量真实样例」的方式演示。

数据准备方面，收集了本专业或相近专业的人才培养方案、课程名称与分类等资料，做了简单整理，为后面导入业务库和上传到 Dify 知识库做准备。完整覆盖所有专业不现实，论文里会说明数据范围。

3. Dify平台与学业知识库

在 Dify 上注册了应用，建了学业相关的知识库，把部分人才培养方案类文档上传并做了分段与向量化，具体参数按 Dify 默认或官方教程微调。

工作流方面，先搭了「查毕业要求」「选课相关咨询」等简单思路，在 Chatflow 里试着接工作流和提示词，并设了一些常用说法，方便用户问得直白时能走到对应流程。

目前属于能跑通演示的程度，回答质量还会受文档是否齐全、模型选择等影响，不追求一次做到完美，后面根据使用情况再改提示词和知识库内容即可。

4. 后端与接口（技术选型：统一采用 FastAPI）

后端全部用 FastAPI 来写，主要因为学校课程和资料里接触较多，语法相对直观，而且自带接口文档（Swagger），写论文和答辩时好截图说明。

选型上还考虑到以后调用 Dify 接口时，用异步写法相对方便，不必一开始就上很复杂架构。

目前已完成项目骨架、用户注册登录、JWT 这类基础接口。学业信息查询、课表、时间规划等接口有的已经写好、有的还在和前端对字段。

Dify 的对话接口在后端做了简单封装，前端可以发问题、拿回复，会话 id 按 Dify 返回保存，方便多轮对话。

整体目标是「能演示完整链路」，不追求高并发、微服务等企业级指标。

5. 前端（Web）

前端按《前端指南》用组件化方式搭页面，先做登录注册和首页导航，再做学号查询页、AI 对话页。

论坛、课表、日历等页面有的已经搭出架子、有的还在做，以结题前能完整走通主要流程为目标。

界面风格以清晰好用为主，不追求复杂动效。系统只在浏览器里使用，做好一般电脑屏幕下的布局，并适当兼顾手机浏览器能点开用，简单响应式即可，不做小程序。

论坛若时间紧，会优先保证发帖、看帖、简单评论，二级评论、点赞等按时间能多做就多做。

6. 阶段性成果归纳

到目前为止，可以演示的大致包括：用户登录后进入首页，查看各功能入口；用测试学号查看学业信息，数据为模拟或与真实结构一致的小样例；打开 AI 助手页面提问，能收到基于知识库或工作流的回答，效果随问题与知识库内容变化。

数据库、Dify、后端、前端的主干已经串起来一大半，和开题报告里「先打基础、再做核心功能」的安排基本一致。

尚未全部完成的部分在第二节说明，后续在第三节写打算怎么补。

7. 与开题报告计划的大致对照（便于老师看进度）

开题报告里第一阶段侧重环境、数据库、数据与 Dify 知识库准备，本人认为这部分基本完成或过半。

第二阶段侧重用户认证、学号查询、AI 问询、论坛、课表与时间规划等模块开发，本人目前认证与学号查询、AI 问询链路相对靠前，论坛与课表、日历仍在推进中。

第三阶段及以后的原定「小程序」已根据本人实际情况取消，改为只做 Web，并在中期报告中统一说明，避免与开题报告个别字句不一致时产生误解，论文里也会写一句「实施范围调整」即可。

二、存在的具体问题

在推进过程中，遇到下面几类问题，有些已经知道怎么改，有些还需要边做边试，答辩时会如实说明，不会夸大。

1. 知识库与数据质量

人才培养方案经常是 PDF、Word，自动提取文字时容易乱版，表格里的学分要求有时对不齐。

知识库里目前只放了部分文档，专业和课程种类不全，AI 回答有时会偏泛或漏细节。

后面打算人工校对重要段落、补几条典型专业的文档，不追求覆盖全校所有专业，够用、能说清楚方法即可。

2. AI回答准确性与可控性

用户问法一复杂，模型偶尔会答偏，或者把不同年级的要求混在一起。

触发词和工作流还在磨合，多轮对话时有时记不住上文，需要在前端或后端把学号、专业等关键信息传稳一点。

错题集、日志分析打算用简单表格或数据库记几条典型错误，有时间就优化提示词和知识库，没时间至少在论文里写清楚局限和改进方向。

3. Dify与业务系统深度集成

Chatflow 里调工作流、再调自己写的数据库接口，调试步骤多，超时或报错时要不要重试、怎么提示用户，还要再统一一下。

API Key 不能写进代码里上传仓库，部署前改成环境变量。调用大模型可能有费用，演示阶段控制调用次数即可。

4. 论坛与文件模块

论坛要发帖、列表、详情、评论，还要上传附件，涉及文件存哪里，例如本机目录或免费额度内的对象存储，以及文件大小限制、格式校验，比纯页面展示麻烦。

若进度紧，可能先做小文件上传或只允许常见格式，审核功能作为可选，不强行上复杂内容审核。

5. 课表与时间规划

若没有学校官方课表接口，课表主要靠导入或手工维护，产品上要写清楚「以用户维护为准」。

日历和课表是否要自动同步，打算先做「一键从课表生成日历事件」或手动添加二选一，提醒用页面内提示即可，不承诺复杂推送。

6. 测试与性能

目前以手工点一点、Postman 调接口为主，单元测试写得不多。

性能方面，毕设演示人数少，先把功能跑通；若论文要写性能，再做一两次简单压测或记录响应时间，不攀比商业系统指标。

三、下一步工作具体设想与安排

下面按时间大致排一下，周数是估计，学校若有统一节点会再调整。原则是先把必交成果做出来，再考虑加分项。

1. 完成核心功能闭环（约2～3周）

优先补全：论坛基本流程，包括发帖、列表、详情、评论，文件上传能有一种稳定方案即可。

课表周视图、学期切换、备注编辑。时间规划日历、事件的增删改。

学号查询各块与接口字段对齐、页面别报错。

不要求每个边角功能都极致，能完整演示一条用户故事即可。

2. 深化Dify与AI质量（约1周，与上面可部分并行）

根据实际问答效果，微调提示词、知识库分段或检索条数。

把明显错误的问答记几条，写进论文「问题与改进」。不承诺准确率达到很高百分比，以「可解释、可改进」为主。

3. Web前端体验与响应式（约1周）

统一各页的加载状态、错误提示。主要页面在手机浏览器里能看清、能点。

检查与 FastAPI 的跨域、Token 过期跳转登录等小问题。

4. 测试、优化与部署（约1～2周）

做一轮功能自测，列简单测试用例表。有时间则对常用接口记一下响应时间。

部署选一种自己搞得定的方式，例如云服务器加 Nginx 反代，或仅本地演示并说明部署方案，以答辩能打开系统为准。

5. 论文与答辩材料（持续）

边做边记：系统结构图、关键接口说明、几张界面截图、遇到的问题与解决办法。

论文按学校模板写，突出「做了什么、怎么做的、局限是什么」，不吹过头。

6. 预期交付物清单（结题前自查用）

可运行的 Web 前端与 FastAPI 后端源码。数据库脚本或说明文档。

Dify 应用与知识库配置说明（脱敏，不含真实密钥）。毕业论文与答辩 PPT。

以上清单按学校最终要求为准，写不下的以附件形式提交。
